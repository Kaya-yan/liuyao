# 文档与工程一致性修复设计

## 背景

当前仓库的核心产品流程已经可用，但文档和工程声明存在几处明显漂移，影响项目可信度与后续协作：

1. `CLAUDE.md` 仅引用 `@AGENTS.md`，但仓库内缺失 `AGENTS.md`，协作入口断链。
2. `README.md` 中有多处描述与实现不一致，包括仓库地址占位符、移动端交互描述、数据存储边界说明、目录结构遗漏现有页面。
3. `package.json` 与 `package-lock.json` 的依赖声明不一致，`framer-motion` 与 `lunar-javascript` 已出现在锁文件中，但未出现在顶层依赖声明中。
4. 仓库缺少最小协作文档，后续协作者和 AI 助手需要从代码中反推项目边界与修改注意事项。

本次工作只修复真实性与可协作性问题，不扩展到业务功能、视觉风格或算法重构。

## 目标

- 修复 `CLAUDE.md` 的断链问题，恢复稳定的项目协作入口。
- 让 `README.md` 与当前实现保持一致，避免对外说明失真。
- 对齐 `package.json` 与 `package-lock.json` 的依赖声明，消除环境描述漂移。
- 增加最小协作文档，使新接手的人可以快速理解项目边界与关键约束。

## 非目标

- 不修改六爻算法、解读规则或八字计算逻辑。
- 不调整页面交互、UI 样式或现有产品文案风格。
- 不清理未使用依赖，不做包升级，不重写目录结构。
- 不处理当前未提交的页面业务改动，除非文档需要准确反映事实。

## 方案选择

采用“最小协作闭环”方案：

- 保留 `CLAUDE.md` 作为极简入口文件。
- 新增 `AGENTS.md` 作为项目协作主说明。
- 修正 `README.md` 中已确认的事实漂移。
- 在不扩大范围的前提下补齐 `package.json` 依赖声明。

未采用的方案：

- 仅做最小真实性修复：范围更小，但无法解决协作入口缺失问题。
- 一次性补齐全部规范文档：范围过大，会偏离本次“最小修复”的目标。

## 当前事实基线

以下内容在开始修复前已经核对：

- 真实远程仓库地址：`https://github.com/Kaya-yan/liuyao.git`
- 当前分支：`master`
- 当前存在未提交改动：`app/cast/page.tsx`、`app/input/page.tsx`、`app/location/page.tsx`、`app/result/page.tsx`、`components/AnalysisLoading.tsx`
- 当前实现包含历史记录页：`app/history/page.tsx`
- 当前结果页会将部分占卜记录写入浏览器本地 `localStorage`：`lib/utils/history.ts`
- 当前移动端默认起卦方式为“摇铜钱”按钮交互，桌面端默认方式为太极图旋转；当前实现没有接入 `devicemotion` 事件：`app/cast/page.tsx`

## 文件级改动设计

### 1. `CLAUDE.md`

处理方式：保留极简入口角色，继续引用 `AGENTS.md`。

设计要求：

- 不把 `CLAUDE.md` 扩写成完整协作文档。
- 让它继续承担统一入口作用，降低后续维护成本。
- 内容保持简短，避免与 `AGENTS.md` 形成双份事实源。

### 2. 新增 `AGENTS.md`

角色：项目协作主说明，面向后续开发者与 AI 助手。

内容范围：

1. 项目一句话目标
2. 本地启动方式
3. 关键目录与职责
4. 数据存储边界
5. 修改时的关键约束
6. 维护原则

具体写法要求：

- 只写当前能从仓库状态确认的事实。
- 不写会快速过期的状态型内容。
- 不复制 README 的长篇产品叙述。
- 明确以下边界：
  - 项目为纯前端 Next.js 应用
  - 无服务端数据库持久化
  - 历史记录会写入浏览器本地 `localStorage`
  - 解读与算法应保持确定性，不应引入随机生成文案

### 3. `README.md`

处理目标：保留现有产品表达风格，同时修正不真实或已漂移的说明。

计划调整项：

#### 3.1 仓库与部署信息

- 将示例 clone 地址从 `https://github.com/user/liuyao.git` 改为真实仓库地址 `https://github.com/Kaya-yan/liuyao.git`
- 将 Vercel 按钮中的仓库地址同步改为真实地址

#### 3.2 功能与实现说明

保留“多源熵值”“真太阳时”“确定性规则引擎”等真实能力描述。

修正点：

- 不再宣称移动端使用 `DeviceMotion` 起卦
- 将相关表述改为当前真实实现：
  - 起卦使用时间、经纬度、交互输入、设备信息与生辰哈希等熵源
  - 移动端当前默认是“摇铜钱”按钮交互
  - 桌面端支持太极图拖拽旋转起卦
- 不写“已接入设备运动传感器”这类当前代码中不存在的能力

#### 3.3 隐私与存储边界

- 不再写“刷新就没了”
- 改为准确说明：
  - 项目没有后端和数据库
  - 用户数据不会上传到服务端持久化
  - 但历史记录会保存在浏览器本地存储中

#### 3.4 技术栈与目录说明

- 技术栈列表与 `package.json` 对齐
- 目录说明补上 `app/history/page.tsx`
- 目录描述只反映当前仓库真实存在的页面和模块

### 4. `package.json`

处理目标：补齐与 `package-lock.json` 一致但缺失的顶层依赖声明。

计划改动：

- 将 `framer-motion` 加回 `dependencies`
- 将 `lunar-javascript` 加回 `dependencies`

本次不做的事：

- 不升级依赖版本
- 不删除锁文件中的包
- 不做未使用依赖清理

这样处理的理由：

- 本次目标是消除工程声明漂移，而不是进行依赖瘦身
- 在未单独审查依赖用途前，补声明比删除更稳妥

## 验证设计

实施后进行以下验证：

1. 文档一致性检查
   - `CLAUDE.md` 能正确指向 `AGENTS.md`
   - `README.md` 的仓库地址、交互描述、数据存储描述与代码一致
   - `AGENTS.md` 不与 `README.md` 形成矛盾的双份说明

2. 依赖声明检查
   - `package.json` 与 `package-lock.json` 的顶层依赖声明恢复一致

3. 基础工程检查
   - 运行 `npm run lint`

4. 变更边界检查
   - 不改动业务实现逻辑
   - 不覆盖当前未提交的页面开发工作

## 风险与控制

### 风险 1：README 仍出现二次漂移

控制方式：

- 只写当前代码中可以验证的事实
- 避免宣称尚未接入的传感器能力或隐私边界

### 风险 2：协作文档写得过重

控制方式：

- `AGENTS.md` 只保留最小可协作信息
- 不引入新的流程规范、发布制度或测试制度文件

### 风险 3：依赖修复误伤当前开发

控制方式：

- 只编辑 `package.json`
- 不重新安装依赖，不重写锁文件

- 新增的 `AGENTS.md`
- 修正后的 `README.md`
- 对齐后的 `package.json`
- 一次基础 lint 验证结果
